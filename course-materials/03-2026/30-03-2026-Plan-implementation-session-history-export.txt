в—Ҹ Validation Report: ADR вҶ” Plan вҶ” assistant-ui Alignment

  I've cross-referenced all ADR docs, the plan, and current assistant-ui/Vercel
  AI SDK documentation from Context7. There are several critical misalignments
  that would cause runtime failures if implemented as currently documented.

  ---
  CRITICAL Issues (would break streaming)

  1. Stream Protocol Format вҖ” COMPLETELY CHANGED

  Our ADR/Plan says (OLD format):
  Content-Type: text/plain;charset=UTF-8
  X-Vercel-AI-Data-Stream: v1

  0:"Hello"\n
  0:" world"\n
  d:{"finishReason":"stop"}\n

  Current Vercel AI SDK v6 uses (NEW SSE format):
  Content-Type: text/event-stream
  x-vercel-ai-ui-message-stream: v1

  data: {"type":"start","messageId":"..."}
  data: {"type":"text-start","id":"msg_123"}
  data: {"type":"text-delta","id":"msg_123","delta":"Hello"}
  data: {"type":"text-delta","id":"msg_123","delta":" world"}
  data: {"type":"text-end","id":"msg_123"}

  Impact: The 0:"text"\n format is the legacy AI SDK v3/v4 format. Current
  AssistantChatTransport expects SSE with JSON-typed messages. Our backend would
   produce output that the frontend can't parse.

  Affected files: ADR-000 В§8, ADR-001 В§5, backend/AGENTS.md, root AGENTS.md,
  Plan tasks BE-5, BE-7, BE-8

  ---
  2. useChatRuntime API вҖ” CHANGED

  Our ADR says:
  useChatRuntime({
    api: `/api/sessions/${sessionId}/messages`,
    initialMessages: [...],
  })

  Current assistant-ui API:
  import { useChatRuntime, AssistantChatTransport } from
  "@assistant-ui/react-ai-sdk";

  const runtime = useChatRuntime({
    transport: new AssistantChatTransport({
      api: `/api/sessions/${sessionId}/messages`,
    }),
  });

  Impact: The api and initialMessages are no longer direct props.
  AssistantChatTransport is now required.

  Affected files: ADR-002 В§3 (ChatRuntime setup), frontend/AGENTS.md, Plan tasks
   FE-5, FE-6

  ---
  3. Request Body Format вҖ” MISMATCH

  Our ADR says backend expects:
  { "content": "user message text" }

  AssistantChatTransport actually sends:
  { "messages": [...], "system": "...", "tools": [...], "threadId": "..." }

  Impact: The backend controller would receive messages[] array (the full
  conversation history) instead of a single content string. Our ChatController
  and ChatRequest DTO are designed for the wrong input shape.

  Affected files: ADR-000 В§6, ADR-001 В§3-4, backend/AGENTS.md, Plan tasks BE-7,
  BE-5

  ---
  4. Response Header Name вҖ” CHANGED

  Our ADR says: X-Vercel-AI-Data-Stream: v1
  Current SDK expects: x-vercel-ai-ui-message-stream: v1

  Affected files: ADR-001 В§5, Plan task BE-7

  ---
  MEDIUM Issues

  5. initialMessages Handling

  The old API allowed passing initialMessages directly to useChatRuntime. The
  new API via AssistantChatTransport may handle this differently вҖ” the docs
  don't show an explicit initialMessages prop on the transport. Session resume
  flow may need a different approach.

  Affected: ADR-002 В§3, В§5, Plan tasks FE-5, FE-6

  6. Backend Must Produce SSE (not plain text)

  Spring Boot's ResponseBodyEmitter can produce SSE, but the format is now
  structured JSON events prefixed with data:. The VercelStreamEncoder utility
  described in Plan task BE-5 needs a complete redesign вҖ” it's not just escaping
   quotes anymore, it's producing typed JSON SSE events.

- Impact: follow-up chat cannot work end-to-end.                                
                                                                                
Evidence:

### 2) Streaming chat request omits required OpenAI model (Critical)            
- `ChatService` builds `ChatCompletionCreateParams` without `.model(...)`.      
- During tests/build, logs show repeated runtime exception:                     
``IllegalStateException: `model` is required, but was not set``.                
- Impact: streaming path fails at runtime even if endpoint is added.            
                                                                                
### 3) Initial analysis ignores configured model and hardcodes provider/model   
(High)                                                                          
- `OpenAIConfig` exposes configurable model bean (`openai.model`).              
- `AnalysisService` hardcodes `openai/gpt-4o-mini` directly instead of using    
injected config.                                                                
- Impact: environment configuration is partially ignored; deployment            
portability/regression risk.                                                    
                                                                                
### 4) вҖңMultimodalвҖқ request is implemented as plain text with data URI (High)   
- `AnalysisService` injects `[Image: data:...base64,...]` into a text message   
instead of using structured image content parts.                                
- Impact: model may treat image bytes as text rather than vision input; token   
bloat/cost spike and poor decision quality.                                     
                                                                                
### 5) Frontend expects wrong API shape for session metadata (High)             
- `SessionResponse` returns `{ session, messages }`.                            
- `ChatView` reads `data.intent`, `data.productName`, `data.orderNumber`        
instead of `data.session.*`.                                                    
- Impact: summary bar data wonвҖҷt load correctly; behavior diverges from API     
contract.                                                                       
                                                                                
### 6) Session resume/message hydration missing in frontend chat runtime (High) 
- PRD/ADR require loading persisted messages for returning users.               
- `ChatView` currently initializes runtime transport only and does not map/pass 
 backend `messages` into `initialMessages` for runtime.                         
- Impact: users with stored session IDs do not get full restored conversation   
in UI.                                                                          
                                                                                
### 7) Backend validation gaps can increase abuse/cost risk (Medium)            
- `SessionController` validates non-empty strings but does not enforce max      
lengths for `orderNumber`, `productName`, `description` at request boundary.    
- DB columns define lengths for some fields, but `description` is unrestricted  
text and is forwarded to LLM.                                                   
- Impact: oversized prompts, unexpected DB errors, and elevated token/cost      
exposure.                                                                       
                                                                                
### 8) Potential PII leakage in logs (Medium)                                   
- `SessionController` and `AnalysisService` log order number and product name.  
- Impact: sensitive customer/order data may leak into centralized logs.         
                                                                                
### 9) Data integrity weakness: no FK/unique ordering constraints on chat       
messages (Medium)                                                               
- `ChatMessage` stores `sessionId` as scalar UUID without entity                
relationship/DB FK constraint.                                                  
- No DB-level uniqueness constraint on `(session_id, sequence_number)`.         
- Impact: orphan records or duplicate ordering possible under                   
concurrency/failures.
